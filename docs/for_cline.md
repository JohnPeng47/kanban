# Request For Employment
## The Future of IDEs
This repo contains a project that I built for applying to a role here at Cline. The original idea for this was, looking back, completely delusional in scope. I had set out to create the "IDE 2.0" prototype -- basically, an alternative editor UX that would use diagrams (generated by Claude) as the main representation of source code instead of .. well, source code. The chain of ideas floating in my head looked something like this:

1. There exists an abstraction *above* source code that is now viable as the main representation for source code because line-by-line understanding of the code is no longer nessescary (with the advent of coding agents)
2. This abstraction is *diagrams generated by Claude Code* because:
a) It can represent and/or hide arbitrary amounts of detail
b) It can do this at a very high spatial density because ascii is layout-free, free-hand coordinate system (as opposed to Mermaid)
3. A mapping from source code lines -> diagram elements will allow canvas operations (single select, rectangle select) to select context for coding agents; this is analogous to highlight -> CTRL+K in AI-enabled IDEs. This enabled code context to be selected directly from diagrams
![demo](docs/assets/kanban.gif)

4. Converting ascii -> SVG with a LLM works pretty well (with caveats)
5. SVG can be used to build a web-based programmatic interface

## Dogfooding
So after spending a week to build out the web-based interface, I started dogfooding on some refactoring tasks. This experimental premise was meant to evaluate two claims:

### Claims Evaluating Diagrams as Code
    a) Weak: Diagrams, used in conjunction with a CLI coding agent, are useful for reading and understanding code
    b) Strong: Diagrams, used in conjunction with a CLI coding agent, are useful for writing code

### Findings
The setup was fairly open-ended and non-rigorous; I made a best-effort attempt to rely on diagrams to understand the structure of the code, and to supply context to the agent (via mechanism described in 3 above). After some testing *my initial exuberance for my vision of the IDE 2.0 was quickly disabused*. Most crucially, using diagrams as a 2D context selection representation does not work well; I found that repeated conversations with the agent to refine a set of code contexts for the edit to be much more preferrable and dynamic than selection over a set of static 2D diagrams. Often, details were missing from the diagram components, requiring the agent to pull context into the *text* conversation. And more often than not, I found myself just directly looking at the source files that the agent pulled into the context, bypassing any need to look at the diagrams. 

Conclusion: Agentic chat is superior for Diagrams for writing code (Strong Claim b).

## The Pivot
Determined to salvage something from failed experiment, I decided to pivot slightly, to target the Weak Claim a). Instead of doing the ascii -> SVG conversion, I simply kept everything in ascii. Then I added a simple feature implemented as a VSCode extension for ascii diagrams: the ability to jump to code references and other diagrams. This I found quite useful for exploring new codebases, can be used within VSCode itself, integrates well with its existing workflow (ie. alt + forward/backwards to traverse jump history), and easy to setup with a project's/team's existing workflows.
